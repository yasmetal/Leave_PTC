# ระบบใบลาออนไลน์ (Leave Request System)

เว็บฟอร์มใบลาภายในองค์กร ใช้งานได้บนมือถือ เขียนด้วย HTML + CSS + JavaScript ล้วน (ไม่มี framework) โฮสต์ฟรีบน GitHub Pages และบันทึกข้อมูลที่พนักงานกรอกลง **Google Sheet** (อยู่ใน Google Drive ขององค์กร) แบบเรียลไทม์ผ่าน Google Apps Script

## โครงสร้างไฟล์

```
leave-form/
├── index.html          หน้าเว็บฟอร์มหลัก
├── css/style.css        สไตล์ (responsive, mobile-first)
├── js/app.js             ตรรกะฟอร์ม + ส่งข้อมูล
├── js/config.js          ใส่ URL ของ Google Apps Script ตรงนี้
├── assets/               โลโก้และไอคอนของ Phattha Packdee (logo.png, favicon.png ฯลฯ)
└── apps-script/Code.gs   โค้ดฝั่ง Backend (วางใน Google Apps Script)
```

## ขั้นตอนที่ 1: ตั้งค่า Google Sheet + Apps Script (เก็บข้อมูลใน Google Drive)

1. ไปที่ [sheets.google.com](https://sheets.google.com) สร้างสมุดงานใหม่ ตั้งชื่อ เช่น `ฐานข้อมูลใบลา`
   (ไฟล์นี้จะถูกเก็บใน Google Drive ของบัญชีที่สร้างโดยอัตโนมัติ)
2. ในสมุดงาน ไปที่เมนู **Extensions (ส่วนขยาย) > Apps Script**
3. ลบโค้ดเริ่มต้น (`function myFunction() {}`) ออกทั้งหมด
4. เปิดไฟล์ `apps-script/Code.gs` จากโปรเจกต์นี้ คัดลอกทั้งหมด แล้ววางแทน
5. กด **Save** (ไอคอนแผ่นดิสก์)
6. กด **Deploy > New deployment**
   - คลิกไอคอนเฟือง เลือกประเภทเป็น **Web app**
   - Description: `Leave Form API` (หรือชื่อใดก็ได้)
   - Execute as: **Me** (บัญชีของคุณ)
   - Who has access: **Anyone**
   - กด **Deploy**
7. ระบบจะขอสิทธิ์ (Authorize access) — เลือกบัญชี Google แล้วกด **Advanced > Go to (ชื่อโปรเจกต์) (unsafe) > Allow** (ข้อความ "unsafe" ปรากฏเพราะสคริปต์ยังไม่ได้ verify จาก Google ซึ่งปกติสำหรับสคริปต์ที่เขียนเอง)
8. คัดลอก **Web app URL** ที่ได้ (รูปแบบ `https://script.google.com/macros/s/xxxxx/exec`)

> หากแก้โค้ดใน Apps Script ภายหลัง ต้องกด **Deploy > Manage deployments > แก้ไข (ไอคอนดินสอ) > Version: New version > Deploy** ทุกครั้ง ไม่เช่นนั้น URL เดิมจะยังใช้โค้ดเก่าอยู่

## ขั้นตอนที่ 2: เชื่อมฟอร์มกับ Apps Script

เปิดไฟล์ `js/config.js` แล้วนำ URL ที่ได้จากขั้นตอนที่ 1 มาใส่แทนที่:

```js
const CONFIG = {
  WEBAPP_URL: "https://script.google.com/macros/s/xxxxxxxxxxxxx/exec"
};
```

บันทึกไฟล์

## ขั้นตอนที่ 3: ทดสอบก่อน deploy

เปิด `index.html` ในเบราว์เซอร์ (ดับเบิลคลิกไฟล์ หรือใช้ Live Server) กรอกข้อมูลทดสอบแล้วกด "ส่งใบลา" จากนั้นเปิด Google Sheet ที่สร้างไว้ ควรเห็นแถวข้อมูลใหม่ในชีตชื่อ `LeaveRequests`

## ขั้นตอนที่ 4: Deploy ขึ้น GitHub Pages

โปรเจกต์นี้ถูก push ขึ้น GitHub repo สาธารณะแล้ว (ดูลิงก์ที่ Claude ให้ไว้ในแชท) เว็บไซต์จะพร้อมใช้งานที่:

```
https://<github-username>.github.io/<repo-name>/
```

หากต้องการเปิดใช้งาน GitHub Pages ด้วยตนเอง (กรณีย้าย repo หรือ fork):
1. ไปที่ repo บน GitHub > **Settings > Pages**
2. Source: เลือก branch `main` และโฟลเดอร์ `/ (root)`
3. กด **Save** รอ 1-2 นาที ระบบจะแสดงลิงก์เว็บไซต์ให้อัตโนมัติ

## การใช้งานบนมือถือ

หน้าเว็บออกแบบแบบ mobile-first (responsive) รองรับการกรอกฟอร์มผ่านเบราว์เซอร์บนมือถือได้ทันที แนะนำให้พนักงาน **เพิ่มลงหน้าจอหลัก (Add to Home Screen)** เพื่อเข้าถึงได้เหมือนแอป:
- **iOS (Safari):** แตะปุ่มแชร์ > "เพิ่มไปยังหน้าจอโฮม"
- **Android (Chrome):** แตะเมนู 3 จุด > "เพิ่มไปยังหน้าจอโฮม"

## ข้อมูลที่จัดเก็บ

ทุกครั้งที่มีการส่งฟอร์ม จะมีแถวใหม่ถูกเพิ่มในชีต `LeaveRequests` ประกอบด้วย: เวลาที่บันทึก, ชื่อ-นามสกุล, รหัสพนักงาน, ตำแหน่ง, แผนก, เบอร์ติดต่อ, ประเภทการลา, วันที่เริ่ม-สิ้นสุด, จำนวนวันลา, เหตุผล, ผู้อนุมัติ และผู้ยื่นคำร้อง (ลงชื่อ)

## หมายเหตุด้านความปลอดภัย

- Web App URL ของ Apps Script ทำหน้าที่เหมือน API endpoint สาธารณะ ไม่ควรฝังข้อมูลลับใดๆ ในฟอร์ม
- หากต้องการจำกัดการเข้าถึงเฉพาะพนักงานในองค์กร ให้เปลี่ยนสิทธิ์การ Deploy Apps Script เป็นเฉพาะผู้ใช้ในองค์กร (Google Workspace) แทน "Anyone"
- ข้อมูลใบลาถือเป็นข้อมูลส่วนบุคคล ควรจำกัดสิทธิ์การเข้าถึง Google Sheet เฉพาะ HR/ผู้ที่เกี่ยวข้อง

## ระบบตรวจสอบตัวเอง (ป้องกันปัญหา "ส่งข้อมูลไม่สำเร็จ")

ระบบมีการตรวจสอบสถานะการเชื่อมต่อ 3 ชั้น เพื่อให้เจอปัญหาได้เร็ว โดยไม่ต้องรอพนักงานแจ้งว่าใช้งานไม่ได้:

**1. ฝั่งหน้าเว็บ (อัตโนมัติ)** — ทุกครั้งที่เปิดหน้าเว็บ จะมีป้ายสถานะใต้หัวข้อ (🟢/🔴) แสดงว่าเชื่อมต่อกับ Apps Script ได้หรือไม่ พร้อมบอกสาเหตุคร่าวๆ ถ้าเชื่อมต่อไม่ได้ และจะเช็คซ้ำอัตโนมัติทุกครั้งที่ส่งฟอร์มแล้วไม่สำเร็จ หรือกดปุ่ม "🔄 ทดสอบการเชื่อมต่ออีกครั้ง" ท้ายหน้า

**2. ฝั่ง Apps Script (manual, เจาะลึกกว่า)** — ในไฟล์ `Code.gs` มีฟังก์ชัน `selfTest()` สำหรับ debug โดยตรงจาก Apps Script editor โดยไม่ต้องผ่านหน้าเว็บเลย:
1. เปิด Apps Script ของโปรเจกต์
2. เลือกฟังก์ชัน `selfTest` จาก dropdown ด้านบน (ข้าง Debug)
3. กด **Run**
4. ดูผลลัพธ์ที่ **View > Logs** (หรือกด Ctrl+Enter) — จะบอกทีละขั้นว่าจุดไหนพัง (เข้าถึงชีตไม่ได้ / สร้างชีตไม่ได้ / doPost error) พร้อมคำแนะนำแก้ไข
5. เช็ค Google Sheet จะมีแถวทดสอบขึ้นต้นด้วย `[SELFTEST]` — ลบทิ้งได้หลังตรวจสอบเสร็จ

**3. Executions log** — ถ้า `doPost` error ตอนพนักงานส่งฟอร์มจริง ข้อความ error จะถูกบันทึกไว้ที่เมนู **Apps Script > Executions** (ไอคอนนาฬิกาด้านซ้าย) ย้อนดูได้ทุกครั้งว่าเกิดอะไรขึ้น

### เช็คลิสต์เมื่อ "ส่งข้อมูลไม่สำเร็จ" หรือ "Failed to fetch"

ไล่ตามลำดับนี้:
1. ดูป้ายสถานะบนหน้าเว็บ (🟢/🔴) ก่อน — ถ้าขึ้น 🔴 จะมีข้อความบอกสาเหตุคร่าวๆ ในตัว
2. เปิด Web App URL ตรงๆ ในเบราว์เซอร์ ควรเห็น `{"status":"ok",...}` ถ้าเห็นหน้า Sign in ของ Google แทน → ไป Deploy > Manage deployments > แก้ "Who has access" เป็น **Anyone** แล้วเลือก **Version: New version** ตอน deploy ใหม่
3. รัน `selfTest()` ใน Apps Script (ดูหัวข้อด้านบน) เพื่อดูว่าโค้ดฝั่ง Backend ทำงานถูกจุดหรือไม่
4. ถ้าเพิ่งแก้ไข `Code.gs` แล้วยังไม่เห็นผล — ต้องกด **Deploy > Manage deployments > ไอคอนดินสอ > Version: New version > Deploy** ทุกครั้ง (แก้โค้ดอย่างเดียวไม่พอ ต้อง deploy เวอร์ชันใหม่เสมอ)
5. เช็คว่าไฟล์ที่วางใน Apps Script editor คือเนื้อหาของ `apps-script/Code.gs` จริง ไม่ใช่ไฟล์อื่น (เช่น `js/app.js`) — ถ้าเห็น error ประเภท `document is not defined` แปลว่าเอาไฟล์หน้าเว็บไปวางผิดที่
